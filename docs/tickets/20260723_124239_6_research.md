# Technical Research: Issue 3.1: Database Schema & Connection Pool Setup (Neon + Drizzle) (Issue #6)

## Executive Summary

This research details the implementation strategy for setting up the core database schema and connection pool using Drizzle ORM and Neon serverless Postgres for the Greg Duck application. The scope encompasses establishing the database connection pool using `@neondatabase/serverless` with WebSocket support for Node/Next.js environments, defining Drizzle schema tables (`users`, `animals`, `encounters`) matching PRD specifications, configuring `drizzle.config.ts`, and adding CLI migration scripts to `package.json`. This issue is fully ready for implementation with no blocking dependencies.

## Technical Challenges

- **WebSocket Initialization in Node.js Runtime**: Neon's serverless `Pool` client relies on WebSockets. When running in a Node.js runtime environment (such as Next.js server actions/API routes or migration scripts), `globalThis.WebSocket` is not defined natively. `neonConfig.webSocketConstructor = ws` must be configured conditionally prior to instantiating `Pool`.
- **Clerk User ID Mapping**: The `users` table uses Clerk-provided user IDs (string format like `user_2...`), which must be modeled as `varchar('id', { length: 255 }).primaryKey()` rather than auto-generated UUIDs, while `animals` and `encounters` use auto-generated UUID primary keys (`uuid('id').defaultRandom()`).
- **Geo-Spatial Data Alignment**: Coordinates are stored as `doublePrecision` floats (`latitude` and `longitude`) along with an H3 spatial index (`varchar('h3_index', { length: 32 })`) at resolution 9.
- **Environment Variable Handling during Migration**: `drizzle-kit` requires access to database connection strings during SQL generation and migration execution. `dotenv` must be loaded in `drizzle.config.ts` so `DATABASE_URL` / `DIRECT_URL` environment variables are properly picked up in CLI contexts.

## Proposed Execution Plan

1. **Phase 1: Dependencies & Configuration**
   - Install production dependencies: `drizzle-orm`, `@neondatabase/serverless`, `ws`.
   - Install development dependencies: `drizzle-kit`, `@types/ws`, `dotenv`.
   - Update [package.json](file:///Users/jamescraig/gh/gregduck/package.json) scripts:
     - `"db:generate": "drizzle-kit generate"`
     - `"db:migrate": "drizzle-kit migrate"`
     - `"db:studio": "drizzle-kit studio"`
   - Create `drizzle.config.ts` configured for `postgresql` dialect, reading `schema: "./lib/db/schema.ts"` and `out: "./drizzle"`.

2. **Phase 2: Database Schema & Connection Implementation**
   - Create `lib/db/schema.ts` defining:
     - `users`: `id` (varchar primary key), `username` (varchar), `avatarUrl` (varchar, nullable), `createdAt` (timestamp default now).
     - `animals`: `id` (uuid primary key default random), `name` (varchar), `species` (varchar), `backstory` (text), `photoUrl` (varchar), `discovererId` (foreign key -> users.id), `h3Index` (varchar 32), `latitude` (doublePrecision), `longitude` (doublePrecision), `createdAt` (timestamp default now).
     - `encounters`: `id` (uuid primary key default random), `userId` (foreign key -> users.id), `animalId` (foreign key -> animals.id), `photoUrl` (varchar), `createdAt` (timestamp default now).
   - Create `lib/db/index.ts`:
     - Configure `neonConfig.webSocketConstructor = ws` when `typeof window === 'undefined' && !globalThis.WebSocket`.
     - Initialize `Pool` from `@neondatabase/serverless` using `process.env.DATABASE_URL`.
     - Initialize and export `db = drizzle({ client: pool, schema })`.
   - Update [lib/index.ts](file:///Users/jamescraig/gh/gregduck/lib/index.ts) to export `db` and schema definitions.

3. **Phase 3: Verification**
   - Write unit tests in `tests/db.test.ts` to verify schema table definitions, column types, foreign key relationships, and export integrity.
   - Run `npm run db:generate` to verify SQL migration files are generated in `./drizzle/`.
   - Run `npm run test` to verify all test suites pass.
