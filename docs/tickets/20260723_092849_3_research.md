# Technical Research: Issue 1.3: Testing Infrastructure Setup (Vitest) (Issue #3)

## Executive Summary

This research covers the setup of the testing infrastructure for the Greg Duck project using Vitest, React Testing Library, and Mock Service Worker (MSW). The project currently has a basic Vitest configuration and a couple of test files passing successfully, but lacks code coverage integration, key run scripts, and robust mocks for external integrations (Clerk, Vercel Blob, and Google Gemini API). This issue is fully ready for implementation.

## Technical Challenges

- **Mocking External APIs**: Stubbing services like Clerk, Vercel Blob, and the Google Gemini API requires different approaches. While Clerk's frontend hooks can be stubbed using `vi.mock('@clerk/nextjs')` and its server methods with `vi.mock('@clerk/nextjs/server')`, mocking Vercel Blob and Gemini API is best handled with Mock Service Worker (MSW) to intercept HTTP traffic or direct module mocking with `vi.mock`.
- **Node vs. JSDOM environments**: To optimize test performance, pure logical/mathematical tests should run in a Node environment to avoid the overhead of `jsdom`, which is configured globally in [vitest.config.ts](file:///Users/jamescraig/gh/gregduck/tests/vitest.config.ts#L8). This can be controlled using inline environment comments in the unit tests.

### Technical Debt / Legacy Code

- **Missing Scripts & Dependencies**: `package.json` lacks `"test:watch"` and `"test:coverage"` scripts, and `@vitest/coverage-v8` is not installed.
- **Lack of Cleanups**: The current setup doesn't automatically call `vi.clearAllMocks()` or `vi.restoreAllMocks()` between tests, which can lead to state leak and inter-test pollution.
