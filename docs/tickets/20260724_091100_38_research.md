# Technical Research: 6.1a: Camera & Geolocation Utility Modules (lib/camera.ts + lib/location.ts) (Issue #38)

## Executive Summary

This technical research pass analyzes the requirements, browser API capabilities, and test architecture for building `lib/camera.ts` and `lib/location.ts` (Issue #38), which serve as the foundational hardware utility modules for the Web Camera & Capture feature. The utilities handle platform-specific quirks (e.g. `OverconstrainedError` fallback chains for back cameras, canvas 1920x1080 resolution bounding, and `GeolocationPositionError` mapping) while remaining pure TypeScript modules isolated from UI frameworks. All tooling, type definitions, and test infrastructure (Vitest + jsdom) are present and verified in the repository, making Issue #38 fully ready for implementation.

## Technical Challenges

- **`OverconstrainedError` Fallback Chain**: Mobile devices and webcams vary significantly in how they interpret `facingMode`. Requesting `{ video: { facingMode: { ideal: 'environment' } } }` may throw `OverconstrainedError` on non-compliant browsers or desktop hardware. A 3-tier fallback strategy (Ideal Environment -> Basic Environment -> Generic Video) is required to ensure reliable camera access.
- **Canvas Memory Bounding & Aspect Ratio Preservation**: Direct frame extraction from high-megapixel cameras (e.g., 4K/8K mobile sensors) can consume excessive browser memory when calling `toBlob()`. Frame capture must dynamically calculate scale factors to bound dimensions to max `1920x1080` while retaining aspect ratio.
- **DOM API Mocking in Vitest/jsdom Environment**: `navigator.mediaDevices.getUserMedia` and `navigator.geolocation.getCurrentPosition` are not fully implemented in `jsdom`. Unit tests must mock these browser interfaces, `MediaStreamTrack`, and `HTMLCanvasElement.prototype.toBlob` cleanly without side effects.
- **Framework Separation Contract**: Acceptance criteria explicitly restrict importing React or Next.js inside `lib/camera.ts` or `lib/location.ts`. Pure TS modularity ensures high testability and reusability.

## Proposed Execution Plan

1. **Phase 1: Implement `lib/camera.ts`**
   - Export custom error `CameraError` with code `'NOT_SUPPORTED' | 'PERMISSION_DENIED' | 'CAPTURE_FAILED'`.
   - Implement `isCameraSupported(): boolean`.
   - Implement `requestCameraStream(): Promise<MediaStream>` with 3-level constraint retry fallback (`{ ideal: 'environment' }` -> `'environment'` -> `true`).
   - Implement `stopCameraStream(stream: MediaStream): void`.
   - Implement `captureFrameAsBlob(videoEl: HTMLVideoElement, quality?: number): Promise<Blob>` with 1920x1080 max resolution bounding and `image/jpeg` compression (default quality 0.85).

2. **Phase 2: Implement `lib/location.ts`**
   - Export interface `GeoCoords` (`latitude`, `longitude`, `accuracy`, `timestamp`).
   - Export custom error `LocationError` with code `'NOT_SUPPORTED' | 'PERMISSION_DENIED' | 'TIMEOUT' | 'UNAVAILABLE'`.
   - Implement `isGeolocationSupported(): boolean`.
   - Implement `getCurrentLocation(options?: PositionOptions): Promise<GeoCoords>` with default options `{ timeout: 10000, maximumAge: 30000 }` and mapping for `GeolocationPositionError` (1 -> `PERMISSION_DENIED`, 2 -> `UNAVAILABLE`, 3 -> `TIMEOUT`).

3. **Phase 3: Unit Testing & Verification**
   - Build `tests/lib/camera.test.ts` covering support checks, stream acquisition, fallback retry chain, stream stopping, and frame canvas blob capture.
   - Build `tests/lib/location.test.ts` covering support checks, location resolution, default options, and error code mappings.
   - Run typecheck `npx tsc --noEmit` and test suite `npm run test`.
