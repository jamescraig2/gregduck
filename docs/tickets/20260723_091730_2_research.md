# Technical Research: Issue 1.2: Establish Code Quality Guardrails (ESLint & Prettier) (Issue #2)

## Executive Summary

This research assesses the requirements and current codebase setup for establishing code quality guardrails (ESLint, Prettier, Husky, and lint-staged) in the `gregduck` repository. The codebase currently lacks formatting configurations (Prettier) and git hooks (Husky, lint-staged), while ESLint is using the modern Flat Config (`eslint.config.mjs`). This issue is fully ready for implementation, and the proposed plan outlines installing the necessary devDependencies, configuring lint-staged/Husky, and extending ESLint with strict TypeScript and unused imports rules.

## Technical Challenges

- **ESLint Flat Config Integration**: ESLint 9 uses Flat Config format (`eslint.config.mjs`). New plugins (like `eslint-plugin-unused-imports`) and configuration extenders (like `eslint-config-prettier`) must be imported using ESM imports rather than the legacy `.eslintrc` string-based `plugins` or `extends` arrays.
- **Next.js Lint-Staged Argument Handling**: Running `next lint` inside `lint-staged` causes issues because `next lint` expects a full project scan by default and can behave unpredictably or slow down when passed individual files. Direct invocation of `eslint --fix` for staged files is the standard solution to guarantee fast and accurate pre-commit linting.
- **Type Checking Staged Files**: Running `tsc --noEmit` on individual staged files fails because TypeScript requires the full context of `tsconfig.json` to properly resolve imports and types. The pre-commit hook must run type-checking globally via `tsc --noEmit` or use `tsc-files` to check modified files correctly. Given the current project size, running a full `tsc --noEmit` is fast and provides 100% type-safety.

### Technical Debt / Legacy Code

- **Prettier Lack of Setup**: No `.prettierrc` or `.prettierignore` exists. Files are formatted manually or not at all.
- **Unused Code and Imports**: No rules are configured to detect and automatically remove unused imports and variables, which leads to dead code in the repository.
- **Missing Hook Infrastructure**: Husky is not initialized and there are no existing git hooks, exposing the codebase to the risk of broken commits.

## Proposed Execution Plan

### 1. Phase 1: Preparation

- Install the required dependencies:
  ```bash
  npm install -D prettier eslint-config-prettier eslint-plugin-unused-imports husky lint-staged
  ```
- Initialize Husky in the repository:
  ```bash
  npx husky init
  ```
- Add a script to `package.json` to ensure Husky hooks are installed on package installation:
  ```json
  "prepare": "husky"
  ```

### 2. Phase 2: Implementation

- **Create `.prettierrc`**: Define standard styling preferences (single quotes, trailing commas, 100-character line limit, etc.).
- **Create `.prettierignore`**: Ignore `.next/`, `node_modules/`, `coverage/`, and build artifacts from Prettier.
- **Update `eslint.config.mjs`**:
  - Import `eslint-plugin-unused-imports` and `eslint-config-prettier`.
  - Add rules for unused imports and strict TypeScript guidelines while ensuring Prettier configurations disable conflicting ESLint formatting rules.
- **Configure `lint-staged`**: Add configuration in `.lintstagedrc.json` to run `eslint --fix` and `prettier --write` on staged files.
- **Configure Pre-Commit Hook**: Set `.husky/pre-commit` to execute type safety checks and then run `lint-staged`:
  ```bash
  #!/bin/sh
  npx tsc --noEmit
  npx lint-staged
  ```

### 3. Phase 3: Verification

- Verify formatting via `npx prettier --check .`.
- Verify linting via `npm run lint`.
- Verify TypeScript compiler diagnostics via `npx tsc --noEmit`.
- Commit changes and confirm that the pre-commit hook runs successfully.
