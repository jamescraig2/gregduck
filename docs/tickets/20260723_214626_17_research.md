# Technical Research: Issue 6.1: Web Camera & Capture UI Component (Issue #17)

## Executive Summary

This research pass evaluates the requirements and technical architecture for building the Web Camera & Capture UI component for Gray Duck. The component must support live video stream initialization via `navigator.mediaDevices.getUserMedia`, fallback photo uploads via file input, client-side HTML5 Geolocation gathering, multipart/form-data POST requests to `/api/capture`, translucent glassmorphism loading overlays, and interactive response cards for discoveries vs. existing encounters. All design system tokens (Issue #24) are verified present in `app/globals.css`, and the `/api/capture` endpoint contract (Issue #16) is fully specified and implemented on backend branches, making this issue fully ready for implementation.

## Technical Challenges

- **Secure Context & Hardware Fallbacks**: Both `getUserMedia` and `navigator.geolocation` require a Secure Context (HTTPS or localhost). On unsupported, insecure, or camera-denied contexts, the UI must seamlessly fall back to `<input type="file" accept="image/*" capture="environment">` without breaking the user workflow.
- **iOS Safari Video Constraints & Frame Extraction**: On iOS Safari, `<video>` elements require `playsinline`, `autoplay`, and `muted` attributes to prevent automatic fullscreen playback. Furthermore, exact camera facing mode constraints (`facingMode: { exact: "environment" }`) will throw `OverconstrainedError` on desktop browsers, requiring progressive constraint fallback.
- **Async State Machine & Geolocation Timeouts**: Geolocation acquisition via `navigator.geolocation.getCurrentPosition` can hang indefinitely if options like `timeout` are omitted. The UI must handle multiple state transitions (Idle, Camera Active, Location Acquired, Gemini Vision Processing, Result Card Modal, Error Toast) while ensuring active video tracks are cleanly stopped to release device camera locks.
- **Canvas Image Blob Conversion**: Offscreen `<canvas>` drawing and export to JPEG Blob must use non-blocking `toBlob()` with proper video aspect ratio and resolution bounds to prevent memory bloat during API transmission.

### Technical Debt / Legacy Code

- **Absence of Dedicated Frontend Components Directory for Features**: Currently `components/ui/` only contains generic visual wrappers (`GlassCard`, `GlassPanel`, `Skeleton`). Interactive feature components (like camera viewfinder, modals, error toasts) need clean modular organization under `components/camera/` or `components/capture/`.
- **Mocking Browser Media API in Testing**: Standard JSDOM environment in Vitest does not mock `navigator.mediaDevices` or `navigator.geolocation`. Unit and component tests must establish clean mock helpers for `getUserMedia`, `MediaStream`, and `getCurrentPosition`.

- **HTML `capture` Attribute**: [MDN Web Docs - capture](https://developer.mozilla.org/en-US/docs/Web/HTML/Attributes/capture) - Mobile camera capture fallback syntax.

## Proposed Execution Plan

1. **Phase 1: Preparation & Utilities**
   - Create media helper module `lib/camera.ts` for camera initialization, constraint fallbacks (`environment` vs `user`), video track lifecycle management (`stopCameraStream`), offscreen canvas frame rendering (`captureVideoFrame`), and file upload parsing (`processFileInput`).
   - Create geolocation helper module `lib/location.ts` wrapping `navigator.geolocation.getCurrentPosition` in a Promise with timeout and accuracy options.

2. **Phase 2: Viewfinder & Capture Component Implementation**
   - Build `components/camera/CameraViewfinder.tsx`: Live video stream container (`<video playsInline autoPlay muted>`), floating shutter button, camera fallback button (`<input type="file" accept="image/*" capture="environment">`), and error notifications.
   - Build `components/camera/CaptureOverlay.tsx`: Translucent glassmorphism loading overlay displaying status steps ("Acquiring location & capturing frame...", "Analyzing animal species with Gemini Vision...").
   - Build `components/camera/DiscoveryModal.tsx` & `components/camera/EncounterModal.tsx`: Interactive result cards for `new_discovery` (emerald theme, species details, rarity badge, action buttons) and `encounter` (amber theme, animal details, distance/encounter update, action buttons).
   - Build composite coordinator component `components/camera/CameraCaptureModal.tsx` managing state transitions and `POST /api/capture` submission.

3. **Phase 3: Verification & Component Testing**
   - Create test suite `tests/components/CameraCapture.test.tsx` using `@testing-library/react` and Vitest, mocking `getUserMedia`, `toBlob`, `geolocation`, and `fetch`.
   - Verify camera fallback, GPS coordinate payload construction, loading states, error toasts, and modal rendering.
