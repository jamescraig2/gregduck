# Implementation Plan: Issue 7 - Clerk Authentication & User Cache Synchronization

## 1. Context

We need to implement user authentication using Clerk (`@clerk/nextjs`), protect routes such as `/capture` and `/journal`, and synchronize user profile data into the local Neon Postgres database.

## 2. File Changes

### 2.1 Dependencies

- Update `package.json` by running `npm i @clerk/nextjs`

### 2.2 Configuration

- Ensure `.env.local` contains `NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY` and `CLERK_SECRET_KEY`

### 2.3 `app/layout.tsx`

- Import `ClerkProvider` from `@clerk/nextjs`
- Wrap the main html/body in `<ClerkProvider>` to supply authentication context globally.

### 2.4 `middleware.ts`

- Create `middleware.ts` in the project root.
- Implement `clerkMiddleware` to protect `/capture(.*)` and `/journal(.*)`.

### 2.5 `lib/auth/sync.ts`

- Create this new file to expose a `syncUser` function.
- It will verify `auth()` from `@clerk/nextjs/server`.
- If a user exists, query the `users` table via drizzle. If found, return.
- If not found, fetch details from `clerkClient().users.getUser(userId)` and insert into the `users` table, returning the new user profile.

## 3. Verification Steps

1. Run `npm run lint` and `npm run build` to ensure the project builds correctly.
2. Ensure the protected routes (`/capture`, `/journal`) redirect to the Clerk sign-in page if unauthorized.
3. Successfully log in with a Clerk user account.
4. Verify that visiting a protected route invokes `syncUser` (e.g. from a Server Component) and accurately caches user data to the Neon DB.

## 4. Rollback Plan

- Revert changes to `app/layout.tsx`.
- Delete `middleware.ts` and `lib/auth/sync.ts`.
- Run `npm uninstall @clerk/nextjs`.

## Critique

- **Dependency**: The plan mentions `npm i @clerk/nextjs`, which is correct.
- **Paths**: `app/layout.tsx` exists. `middleware.ts` and `lib/auth/sync.ts` will be created.
- **Framework specific**: `middleware.ts` in Next.js root is the correct pattern. `ClerkProvider` wrapping `app/layout.tsx` is also the recommended integration pattern.
- **Database logic**: Integrating with Drizzle via `users` table cache synchronization in `syncUser` makes sense.

**Conclusion**: PASS.
